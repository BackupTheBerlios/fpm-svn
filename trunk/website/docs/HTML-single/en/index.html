1.1. About this library

1.1. About this library

This is a macro library written in the C programming language, and implements fixed point arithmetic. Although current state-of-the-art FPU's are getting faster and faster, it might still be beneficial to implement your algorithms with fixed point arithmetic to gain even higher speeds, although at the cost of precision. Fixed point arithmetic is less precise than floating point arithmetic, but lots of times, the extra precision is not needed.

I chose to implement this library as a set of macro's instead of functions, to avoid having the extra overhead of function calls. This might add a little to the codesize of the program that utilizes this library, but assures you that no precious clockcycles are wasted by saving and restoring CPU registers.

Possible fields of use are:

  • Games
  • Embedded systems
  • Audio and video filtering
  • Optimized codecs

2.1. System requirements

Basically, all you need is a decent C-compiler. If your compiler doesn't accept the -MM command line option for generating make dependencies, you need to have makedepend installed. Optionally, if you have indent on your system, it is used to process the generated LUT header file.

Running benchmarks does not require any other tools, but if you automatically want to generate barcharts of the results, you need gnuplot, awk and fig2dev with PNG support (which requires libpng).

For building your own HTML documentation from the XML files that are in CVS, you need docbook, xmllint, xsltproc and the libraries they depend on, like libxml and libxslt.

All configure scripts and Makefiles assume that basic programs like sh, sed and cut are available.

2.2. Compilation and installation

Most of the time, the following will work fine: 

        ./configure
        make
        make install
Configure will figure out what tools are available on your system and how to use them. Running make will build the final macro library header files and make install will install them in the default installation directory, which usualy is /usr/local/include. After that, you can start developping your program and use this library.

If this doesn't work, or you want a more fine grained control of the build and installation process, you should run: 

        ./configure --help
The next section will describe the configure options in more detail.

2.3. Tweaking the build process

By default, gcc is the C-compiler that's used. You can override that by either setting the CC environment variable or by using the --cc command line option of configure. 

        CC=/usr/local/intel/bin/icc ./configure 
        ./configure --cc=tcc

You can override the default CFLAGS by setting the environment variable of the same name. 

        CFLAGS=-time ./configure
If CFLAGS is set by default in your environment and you don't want that, you can specify an empty CFLAGS= on your command line.

You can also specify extra directories to look for include files and libraries. If your compiler can't find math.h by default, or fails to link against libm, you can use: 

        ./configure --with-extraincdir=/opt/some/dir/include 
        ./configure --with-extralibdir=/opt/some/dir/lib

If certain applications needed for the build process can't be found in your default PATH, but you are sure they are installed on your system, you can pass the absolute pathnames to configure: 

        ./configure --with-makedepend=/not/in/my/path/bin/makedepend
Similar options are available for all applications that might be needed for the basic build process or for running benchmarks and generating barcharts. Check ./configure --help for a list of all available options.

If you plan to work on the library itself, maybe enhance its functionality or improve the algorithms that are currently in use, you can enable debug symbols and/or all compiler warnings: 

        ./configure --enable-debug 
        ./configure --enable-warnings

First, your application needs to include the Fixed Point Math Macro Library header file: 

        #include <fixedpointmath.h>
After that, you will be able to use its types and functions. There's no need to link to any special libraries.

3.1. Basic types

There are four basic fixed point types. They differ in the amount of memory they occupy and the amount of precision they offer. All four come in both signed and unsigned variants, which sums up to a total of eight types.

Table 3.1. Basic types

type total fraction min max precision
fp8p8_t 16 8 -128.999... 127.999... 0.00390625
fp24p8_t 32 8 -8388608.999... 8388607.999... 0.00390625
fp16p16_t 32 16 -32768.999... 32767.999... 0.00001526
fp8p24_t 32 24 -128.999... 127.999... 0.00000006
ufp8p8_t 16 8 0.0 255.999... 0.00390625
ufp24p8_t 32 8 0.0 16777215.999... 0.00390625
ufp16p16_t 32 16 0.0 65535.999... 0.00001526
ufp8p24_t 32 24 0.0 255.999... 0.00000006

Declaring variables of certain types is as simple as with any type: 

        ufp16p16_t x, y;
        fp16p16_t matrix[16][16];

3.2. Initialization and type conversion

To initialize your variables, you need to convert a value from another type and assign it to your fixed point variable. For each fixed point type, there are several macro's that convert to and from other types.

Table 3.2. Type conversion to fp8p8_t

from fp8p8_t
int itofp8p8(x)
float ftofp8p8(x)
double dtofp8p8(x)

Similar macro's exist for all other fixed point types.

Example: 

        int        a;
        ufp16p16_t b;

        a = 97;
        b = itoufp16p16(a);

        [...]

        b = itoufp16p16(123);

There are also macro's that convert your fixed point variables back to standard types.

Table 3.3. Type conversion from fp8p8_t

from int float double
fp8p8_t fp8p8toi(x) fp8p8tof(x) fp8p8tod(x)

Again, similar macro's exist for all other fixed point types.

Example: 

        fp16p16_t a;
        float     f;

        [...]

        f = fp16p16tof(a);

Finally, there are macro's to convert between fixed point types. Their names are structured analog to the above macro's. Note that there are no explicit macro's to convert between signed and unsigned types. If you want to do that, you have to know exactly that what you are doing will work correctly.

Example: 

        fp24p8_t   a;
        fp8p8_t    b;
        ufp16p16_t c

        [...]

        a = fp8p8tofp24p8(b);

        /* incorrect if b is negative */
        c = ufp8p8toufp16p16(b);

3.3. Doing calculations

For addition and substraction, all fixed point variables can be treated as if they were normal variables.

Example: 

        fp16p16_t x, y, z;

        [...]

        z = x + y;
        z = x - y;

Multiplication and division are a bit more complicated. Using the basic '*' and '/' operators will not work with fixed point variables. Since C doesn't support overloading of operators, there are multiple macro's that do the job. One for each type. Macro's for multiplication start with 'mul' and those for division start with div. All macro's take two arguments.

mul...(x, y) = x * y

div...(x, y) = x / y

Both arguments have to be of the same type and can be other variables, constants or the same variable.

Example: 

        fp8p8_t   x, y, z;
        ufp8p24_t a;

        [...]

        z = mulfp8p8(x, y);
        x = mulfp8p8(x, itofp8p8(15));
        y = divfp8p8(x, z);

        a = mulufp8p24(a, a);

3.4. sqrt, ln, exp and pow

For all fixed point types, the macro library implements sqrt, ln, exp and pow.

Table 3.4. Math macro's for ufp16p16_t

  description
sqrtufp16p16(x) calculates the square root of x for x >= 0
lnufp16p16(x) calculates the natural logarithm of x for x >= 0
expufp16p16(x) calculates e to the power of x for x >= 1
powufp16p16(x, y) calculates x to the power of y for y >= 1

For all seven other types, similar macro's exist. Also, be aware that the exp and pow macro's easily overflow your variables!

Example: 

        fp24p8_t a;

        a = ftofp24p8(2.37);

        a = sqrtfp24p8(a);

4.1. Look Up Tables

By default, all mathematical functions are calculated and/or approximated algorithmicly. In some cases the algorithms use a few (static) temporary variables, but generally speaking, they don't use large amounts of memory and in most cases they are faster than their floating point equivalents.

In some cases though, they are not. For example, Athlon CPU's "cheat" a little by using a 14-bit look-up table for square root calculation and are therefore actually faster! This is clearly not what we want.

This Fixed Point Math Macro Library also supports look-up table variants of all mathematical functions. But considering its fields of use, it has to be cache friendly. Large tables would ruin the L1 cache. So instead of one large table, I chose to use several small tables and linear interpolation for the values that don't directly map to the look-up table. Depending on the mathematical function that's approximated and the precision that's needed, different tables are used to accomplish that. For example, the square root function needs more precision for small values of x, but once the "curve" is mostly done, it's almost a straight line and less samples are needed to quite accurately approximate the function. Depending on the fixed point type chosen and the function at hand, two to four tables are needed, each consisting of somewhere around 128 values. This assures only a few kilobytes of memory and hence won't trash the cache in most cases.

Look-up tables can be activated by using #define before including fixedpointmath.h or by using the -D command line option of your compiler. Currently (as of the alpha release) the following defines are supported:

__FPM_ENABLE_ALL_LUT__ Enable look-up tables for all calculations
__FPM_ENABLE_SQRT_LUT__ Enable look-up tables for all square root calculations

Eventually, all mathematical functions will have their look-up table equivalent. I even consider splitting it up for every basic fixed point type, which could save memory. Embedded systems with only limitted amount of RAM or ROM, could benefit from that.

Example: 

        #define __FPM_ENABLE_SQRT_LUT__
        #include <fixedpointmath.h>

        fp16p16_t a, b;

        [...]

        a = sqrtfp16p16(b);     /* will be "looked up" and possibly
                                 * interpolated */
